Reading Decoded Data

Applications can receive decoded data either via the keyboard input stream in “Wedge mode”, or by posting requests to the decoding driver.

 

Keyboard Wedge

Each data type (with the exception of image data) has an ability to be integrated into the keyboard input stream, allowing applications which are not aware of decoding to still receive decoded data. Decoded input will be seen by applications in precisely the same manner as keyboard input. This feature is independently selectable for each data type. This means that if two drivers interpreted barcode data, the keyboard wedge setting for barcode input would govern them both. By default, the keyboard wedge feature is enabled, which allows the device to be used immediately to scan barcode labels into applications such as WordPad.

 

Posting Requests

Applications can receive decoded data by posting a request for data to a decoding driver directly. This mechanism gives the greatest level of control over the decoded data stream. Multiple requests may be posted to a decoding driver. Posted requests have the following attributes:

 

Number of Data Samples

Each request specifies from 1 to 255 data samples (labels, MSR swipes, etc.) to collect. The application posting the request is notified as each data sample is collected. When all of the requested number of samples have been processed, the posted request will be removed.

 

Recurring Requests

Posted requests may also be marked as recurring requests, which will treat the number of data samples specified as infinite. This has the side effect of effectively granting the posting application exclusive access to the decoding device.
